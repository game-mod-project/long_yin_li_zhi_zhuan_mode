# v0.7.2 — ItemData 등급·품질 spike 결과

**일시**: 2026-05-03
**dump 경로**: BepInEx/LogOutput.log 의 `[v0.7.2 spike]` 줄
**샘플**: 인벤토리 첫 3 item (Book 타입 / itemLv=5 / rareLv=5)
**전체 인벤 갯수**: 180

## 발견된 필드

| 의미 | 필드명 | 타입 | 값 form | 매핑 |
|---|---|---|---|---|
| 카테고리 | `type` (`ItemType` enum, e.g. `Book`/`Equipment`/`Med`/`Treasure`/`Material`/`Horse`) + `subType` (`Int32`) | int+int | 0..N | `$"{(int)type:D3}.{subType:D3}"` 합성 |
| 이름 | `name` | string | 한자 raw (예: `多情飞刀`, `因陀罗杵`, `仙娥折梅`) | 그대로 |
| 등급 | `itemLv` | Int32 | 0..5 (가설) | 0=열악(劣), 1=보통(普), 2=우수(优), 3=정량(精), 4=완벽(完美), 5=절세(绝世) |
| 품질 | `rareLv` | Int32 | 0..5 (가설) | 0=잔품(残), 1=하품(下), 2=중품(中), 3=상품(上), 4=진품(珍), 5=극품(极) |

## 추가 dump 정보

- 모든 ItemData 는 sub-data wrapper 들 (`equipmentData` / `medFoodData` / `bookData` / `treasureData` / `materialData` / `horseData`) 을 노출하며 해당 카테고리에 맞는 1개만 non-null. 이는 `Type` 구분과 일치 — 향후 카테고리별 추가 reflection 시 활용.
- `value` (Int32) = 가격. Sample 3개 모두 32000 (Book 고가 baseline).
- `weight` (Single) = 무게 (kg) — v0.7.1 ItemListReflector 와 일치.
- 추가 필드: `checkName`, `describe` (string, 비어있음 — 게임 로직 내부 검증용 추정), `isNew` (bool), `poisonNum` (Single), `poisonNumDetected` (bool), `setName` (string).
- IL2CPP wrapper 메타: `ObjectClass`, `Pointer`, `WasCollected`, `isWrapped`, `pooledPtr` — reflection 으로 노출되나 본 모드 사용 안 함.

## 한계 / 검증 필요

- 샘플이 모두 Book + 최대치 → 다른 등급·품질 분포 미확인. **Task 9 smoke 에서 검증**: ContainerPanel 정렬 등급/품질 dropdown 결과로 `0..5` 각 tier 가 색상별로 구분되어 표시되는지 (배경 회색=0 → 빨강=5, 마름모 회색=0 → 빨강=5).
- 0-indexed vs 1-indexed 미확인. 코드는 0-indexed 가정. 만약 1-indexed (1..6) 라면 sort 순서는 그대로 동작 (단, color 매핑이 1-indexed 이면 v0.7.4 Item 상세 panel 시점에 분리 필요).
- enum string 노출 path (Il2CppEnum.ToString()) 는 본 dump 에서 미발견 (모든 grade/quality 가 Int32). 향후 게임 update 가 enum string 으로 바뀌면 ItemReflector 의 GradeMap/QualityMap dictionary 가 fallback 으로 작동.

## ItemReflector 갱신

`Core/ItemReflector.cs`:
```csharp
private static readonly string[] GRADE_NAMES   = new[] { "itemLv", "grade", "level", "lv", "tier", "rank" };
private static readonly string[] QUALITY_NAMES = new[] { "rareLv", "quality", "purity", "pin", "pinji", "pinzhi" };
```

`itemLv` / `rareLv` 가 array 의 첫 번째 — IL2CPP runtime 에서 즉시 hit. 후순위 best-guess 들은 게임 update 등으로 필드명이 바뀔 경우 fallback 으로 남김.

## 후속 sub-project 자산

- D-1 (v0.7.4 Item 상세) 시 `equipmentData` / `bookData` / `treasureData` 등 sub-wrapper 의 추가 reflection 필요. 본 dump 가 wrapper presence 확인 path 제공.
- D-4 (v0.7.5 한글화) 는 `name` 필드 (한자 raw) 가 현 baseline. 한글 패치 mod 의 사전을 hook 하거나 itemID→한글 자체 사전 구축.
- 정렬·검색 hot path 에서 `itemLv`/`rareLv` 가 IL2CPP getter 1회 호출 (~수 ns) → 180 items × 2 reads × 60 fps 로도 < 0.1 ms / frame. cache hit path 에선 zero — v0.7.2 ContainerView 의 cache 가 적절.
